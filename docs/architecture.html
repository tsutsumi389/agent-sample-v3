アーキテクチャ

アーキテクチャ設計

本システムは オーケストレータ + プランナー / エグゼキューター(並列) / エヴァリュエーター 構成の AI エージェントである。LangGraph の親グラフが全体を制御し、各エージェントは独立コンパイルされた StateGraph として動作する。本章では全体構成、親グラフのノード/エッジ、コンテキスト分離、並列実行、State スキーマを示す。

1. 全体構成

ユーザーの入力は React フロントから SSE 経由で FastAPI に届き、LangGraph 親グラフ(オーケストレータ)が処理を統括する。Planner が計画を立て、Executor が並列実行され、Evaluator が結果を評価する。最終回答生成時には背景で会話を記憶化し、PostgreSQL(pgvector)へ永続化する。

  ┌────────┐
  │ ユーザー │
  └───┬────┘
      │ React + SSE
      ▼
  ┌─────────────────────────────┐
  │ FastAPI   /chat/stream      │
  └───┬─────────────────────────┘
      │ astream(stream_mode=["custom","updates"], subgraphs=True)
      ▼
  ┌──────────────────────────────────────────────────────────┐
  │ LangGraph 親グラフ (オーケストレータ)                       │
  │                                                            │
  │  recall_memory → planner ─┬─[Send]→ executor(t1) ┐         │
  │                           ├─[Send]→ executor(t2) │ 並列     │
  │                           └─[Send]→ executor(t3) ┘         │
  │                                       │ fan-in             │
  │                                       ▼                    │
  │                                  evaluator                 │
  │                                  ┌──┴──┐                   │
  │                          合格/上限    不合格                 │
  │                            │            │                  │
  │                            ▼            ▼                  │
  │                        finalize    planner(再計画)          │
  └────────────────────────────┬───────────────────────────────┘
                               │ remember_async(背景で記憶化 / LangMem)
                               ▼
                ┌───────────────────────────────────────┐
                │ PostgreSQL + pgvector                  │
                │ nomic-embed-text (768次元)             │
                └───────────────────────────────────────┘
NOTE 記憶化 (remember_async) は finalize ノードの中でホットパスをブロックせずに背景実行される。DB が未起動の場合は記憶検索/保存がスキップされ、チャット自体は動作する。

2. LangGraph 親グラフ — ノードとエッジ

backend/app/orchestrator.py の build_orchestrator() が親グラフを構築する。StateGraph(ParentState) に 5 つのノードを登録し、条件エッジで Send による並列ファンアウトと評価後ルーティングを実現している。

   START
     │
     ▼
  recall_memory          記憶を想起 (retrieved_memories, iteration=0)
     │  add_edge
     ▼
  planner                計画を生成 (plan)
     │  add_conditional_edges(assign_executors, ["executor"])
     │  ── plan の各タスクを Send で並列ファンアウト ──
     ▼ ▼ ▼
  executor × N           各タスクを実行 (task_results を operator.add でマージ)
     │  add_edge  (fan-in バリア: 全 executor 完了後に評価へ)
     ▼
  evaluator              結果を評価 (evaluation, iteration+1)
     │  add_conditional_edges(route_after_eval, {replan, finalize})
     ├─ replan ───────────► planner   (不合格 かつ iteration < MAX_ITERATIONS)
     │
     └─ finalize          (passed=True または iteration >= MAX_ITERATIONS)
            │  add_edge
            ▼
           END
ノード名関数役割主な書き戻しキー
recall_memoryrecall_memory長期記憶を想起retrieved_memories, iteration
plannerplanner_nodePlanner サブグラフを呼ぶブリッジplan
executorexecutor_nodeExecutor サブグラフを呼ぶブリッジ(並列起動)task_results
evaluatorevaluator_nodeEvaluator サブグラフを呼ぶブリッジevaluation, iteration
finalizefinalize結果を統合し最終回答を生成、背景で記憶化final_answer

分岐は 2 つの関数が担う。assign_executors(state) は plan の各タスクから Send("executor", ...) のリストを返し、route_after_eval(state) は evaluation["passed"] または iteration >= settings.max_iterations なら "finalize"、そうでなければ "replan"(= planner へ戻る)を返す。

3. コンテキスト分離

各サブエージェント(Planner / Executor / Evaluator)は 独立してコンパイルされた StateGraph であり、親グラフと State キーを一切共有しない。オーケストレータの各ブリッジノードは、サブエージェントへ 明示的な指示文だけ を渡し、戻り値の必要なキーだけを親 State に読み戻す。

NOTE State キーを共有すると暗黙の共有チャネルとなり分離が壊れる(schemas.py 冒頭の設計意図)。そのため ParentState と各サブエージェントの State は意図的にキーを分けている。会話履歴は引き継がれず、各エージェントは毎回まっさらなコンテキストで起動する。

具体的にブリッジノードが渡す内容は以下の通り。

  • planner_node → Planner には request / memories / feedback のみ渡す(feedback は前回評価の evaluation.feedback)。
  • executor_node → Executor には instruction / task_id のみ渡す(Send のペイロード由来)。
  • evaluator_node → Evaluator には request / results のみ渡す。

4. 並列実行

Planner が出力したタスク(plan)を、オーケストレータが Send API で並列ファンアウトする。add_conditional_edges("planner", assign_executors, ["executor"]) により、planner ノードの後で assign_executors が Send のリストを返し、各タスクが個別の executor 起動になる。

def assign_executors(state: ParentState):
    """plan の各タスクを並列 Executor として Send でファンアウトする。"""
    plan = state.get("plan") or []
    writer = safe_writer()
    writer({"type": "status", "node": "orchestrator", "state": "running",
            "message": f"{len(plan)}件のタスクを並列実行へ振り分け"})
    return [
        Send("executor", {"instruction": t["instruction"], "task_id": t["id"]})
        for t in plan
    ]

複数の executor が並列に書き込む結果は、State 側のリデューサで安全にマージされる。ParentState.task_results に Annotated[list[TaskResult], operator.add] を付けることで、並列書き込みが結合される(リデューサが無いと InvalidUpdateError になる)。

class ParentState(TypedDict, total=False):
    ...
    task_results: Annotated[list[TaskResult], operator.add]  # 並列マージ
    ...

executor → evaluator の通常エッジは fan-in バリアとして機能し、全 executor が完了してから evaluator が起動する。

制約 単一 Ollama インスタンスへの並列 Executor は、OLLAMA_NUM_PARALLEL 次第でモデルサーバ側で直列化されうる(README「既知の制約」)。

5. State / スキーマ構造

backend/app/schemas.py はドメイン型、親グラフ State、各サブエージェントの独立 State を定義する。すべて TypedDict で表現される。

5.1 ドメイン型

型フィールド説明
Taskid: str, instruction: str, parallel_group: intPlanner が生成するタスク
TaskResulttask_id: str, instruction: str, result: strExecutor の実行結果
Evaluationpassed: bool, feedback: strEvaluator の評価結果

5.2 親グラフ State (ParentState)

フィールド型説明
user_requeststrユーザーの元の要求
user_idstr記憶の名前空間に使うユーザー識別子
retrieved_memorieslist[str]想起された長期記憶
planlist[Task]Planner が生成した計画
task_resultsAnnotated[list[TaskResult], operator.add]並列 Executor の結果(リデューサでマージ)
evaluationEvaluationEvaluator の評価
iterationint評価→再計画ループの回数
final_answerstrfinalize が生成した最終回答

5.3 各サブエージェントの独立 State

親とキーを共有しない、各エージェント専用の State。いずれも TypedDict, total=False。

PlannerState

  • request: str
  • memories: list[str]
  • feedback: str
  • plan: list[Task]

ExecutorState

  • instruction: str
  • task_id: str
  • result: str

EvaluatorState

  • request: str
  • results: list[TaskResult]
  • evaluation: Evaluation

補助関数 empty_evaluation() は {"passed": True, "feedback": ""} を返し、評価が取得できなかった場合のフォールバック(evaluator_node で使用)となる。
