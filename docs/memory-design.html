長期記憶設計

本章では、会話をまたいでユーザーに関する知識を蓄積・想起する長期記憶サブシステムを解説する。記憶は LangMem によって抽出され、pgvector を有効化した PostgreSQL に AsyncPostgresStore 経由で永続化される。読み出しはオーケストレータの recall_memory ノードで、書き込みは応答生成後の背景タスクで行われ、両者は nomic-embed-text(768 次元)の埋め込みによる意味検索で結ばれる。

1. 長期記憶アーキテクチャ

記憶のライフサイクルは「想起(recall)→ LLM 応答 → 背景での記憶化(LangMem)→ 永続化」という流れで構成される。永続化レイヤは AsyncPostgresStore を介した PostgreSQL + pgvector であり、埋め込みは nomic-embed-text による 768 次元ベクトルである。

   ┌──────────────────────── Orchestrator (親グラフ) ────────────────────────┐
   │                                                                         │
   │   START                                                                 │
   │     │                                                                   │
   │     ▼                                                                   │
   │  recall_memory ──── recall_memories(user_id, query) ──┐                 │
   │     │                                                 │  asearch        │
   │     ▼                          retrieved_memories     ▼  (意味検索)     │
   │  planner → executor×N → evaluator → finalize     ┌─────────────────┐    │
   │                                       │           │ AsyncPostgres   │    │
   │                                       │ answer     │ Store           │    │
   │                                       ▼           │   namespace =    │    │
   │                              remember_async(...)   │ ("memories",    │    │
   │                                       │            │   user_id)      │    │
   │                                       ▼            └────────┬────────┘   │
   │                          ReflectionExecutor.submit          │           │
   │                          (背景スレッド / after 0.5s)         │ aput      │
   └──────────────────────────────────────────────────│─────────┼───────────┘
                                                       │         │
                          埋め込み: nomic-embed-text    ▼         ▼
                          (768次元) ───────────►  ┌─────────────────────────┐
                                                  │ PostgreSQL + pgvector    │
                                                  │ (docker-compose db)      │
                                                  │  vector(768) で類似検索   │
                                                  └─────────────────────────┘

想起(read path)

recall_memory ノードが recall_memories() を呼び、store.asearch で関連記憶を取得して retrieved_memories として State に書き込む。これが planner に渡る。

記憶化(write path)

finalize が応答生成後に remember_async() を呼ぶ。ReflectionExecutor.submit が背景で会話から記憶を抽出・統合し、ストアへ aput する。

永続化(store)

AsyncPostgresStore が pgvector 上に記憶を保存。プロセスを再起動しても記憶は残り、名前空間 ("memories", user_id) でユーザー単位に分離される。

2. AsyncPostgresStore の設定・接続・初期化

ストアは backend/app/memory.py で langgraph.store.postgres.aio.AsyncPostgresStore として構築される。非同期コンテキストマネージャであるため、ストア本体 store と CM _store_cm をモジュールグローバルに分けて保持し、アプリのライフサイクル(main.py の lifespan)に合わせて手動で enter/exit する。

_store_cm = AsyncPostgresStore.from_conn_string(
    settings.database_url,
    index={
        "dims": settings.embed_dims,   # nomic-embed-text = 768 固定
        "embed": get_embeddings(),
        "fields": ["text"],
    },
)
store = await _store_cm.__aenter__()
await store.setup()
項目値 / 出典説明
接続文字列settings.database_url既定 postgresql://agent:agent@localhost:5432/agent(config.py)。docker-compose の db サービスに対応。
埋め込み次元settings.embed_dims = 768index.dims に渡され、ベクトル列の次元を確定する。
埋め込みモデルget_embeddings()nomic-embed-text。index.embed に渡され検索/保存時に使われる。
索引対象フィールドfields: ["text"]記憶レコードの text フィールドをベクトル化対象とする。
初期化と破棄

setup_store() は接続プール確立後に store.setup() を実行し、pgvector 拡張の有効化とテーブル作成(IF NOT EXISTS)を行う。冪等なので再起動時の再実行も安全。teardown_store() は CM を __aexit__ し、store / _store_cm / リフレクション関連のグローバルを None にリセットして接続プールを破棄する。

3. 記憶の検索(recall_memories)と保存(背景タスク)

3.1 検索: recall_memories

recall_memories(user_id, query, limit=5) は名前空間 ("memories", user_id) に対して store.asearch(query=...) を実行し、ヒットした各レコードの value から text →(無ければ)content →(さらに無ければ)value 自体の順でテキストを取り出してリスト化する。オーケストレータの recall_memory ノードがこれを呼び、SSE に memory イベントと取得件数の status を流したうえで retrieved_memories として State に書き込む。

3.2 保存: remember_async と ReflectionExecutor

記憶化はホットパスをブロックしないよう背景で行う。finalize が remember_async(user_id, user_message, assistant_message) を呼ぶと、user / assistant の 2 メッセージを payload に組み立て、ReflectionExecutor.submit(payload, after_seconds=0.5, config={"configurable": {"user_id": ...}}) で背景投入する。

  1. _get_reflection() がリフレクション実行器を遅延初期化する(初回呼び出しで一度だけ構築、_reflection_ready フラグで管理)。
  2. store が未初期化なら構築せず None を返し、次回に再試行する。
  3. 構築時は create_memory_store_manager(get_chat_llm(streaming=False), namespace=NAMESPACE, store=store) でマネージャを作り、ReflectionExecutor(manager, store=store) を生成する。
  4. submit 後、背景で会話から記憶が抽出・統合され、同じ名前空間にベクトル付きで保存される。
名前空間の一致

記憶の名前空間は NAMESPACE = ("memories", "{user_id}") として定義され、書き込み(create_memory_store_manager の namespace)と読み出し(recall_memories の ("memories", user_id))で完全に一致させる必要がある。一致しないと保存した記憶が検索でヒットしない。

4. pgvector によるベクトル検索と次元の固定

意味検索は pgvector の vector 型カラムに対する類似度検索で実現される。検索時はクエリ文字列を nomic-embed-text でベクトル化し、保存済みの記憶ベクトルとの近傍を求める。index.dims がベクトル列の次元を確定するため、埋め込みモデルの出力次元と一致していなければならない。

本システムは embed_dims = 768 を nomic-embed-text の固定次元として扱う。起動時の健全性チェック assert_embed_dims() が実際の埋め込み次元(embed_query("dimension probe") の長さ)を計測し、settings.embed_dims と不一致なら RuntimeError を送出する。

def assert_embed_dims() -> int:
    dim = len(get_embeddings().embed_query("dimension probe"))
    if dim != settings.embed_dims:
        raise RuntimeError(
            f"埋め込み次元が想定と不一致: 実際={dim}, 期待={settings.embed_dims}. ..."
        )
    return dim
次元固定の理由

コメントによれば、古い/壊れた埋め込みモデルのタグだと 8192 を返すことがあり、その場合に意味検索が「静かに壊れる」(エラーにならず精度だけ劣化する)。これを起動時に検知するため、768 を期待値として明示的にアサートしている。テスト test_embed_dims_is_768 も assert assert_embed_dims() == 768 でこの不変条件を固定している。

5. DB 未起動時のフォールバック

長期記憶は補助機能であり、PostgreSQL が起動していなくてもチャット自体は継続できるよう設計されている。記憶の検索・保存はスキップされ、空の結果や no-op に縮退する。

状況挙動
store が None(未初期化)で recall_memories 呼び出し即座に空リスト [] を返す。
store.asearch が例外送出warning ログを出し、空リスト [] を返す(検索失敗で会話は止めない)。
store 未初期化で記憶化(_get_reflection)リフレクション実行器を構築せず None を返し、remember_async は何もせず return する。
LangMem マネージャ初期化失敗warning ログを出し _reflection = None とする(記憶化スキップ)。
remember_async の submit 失敗warning ログを出して握りつぶす(背景記憶化失敗で応答は返す)。
縮退時の方針

DB 未起動・接続失敗時はいずれも例外を握りつぶして warning ログにとどめ、recall_memory は 0 件の記憶を取得、finalize は記憶化を no-op としてチャットフローを完走させる。記憶機能の喪失とチャットの可用性を切り離している。

6. docker-compose の構成

永続化先 PostgreSQL は docker-compose.yml の db サービスとして定義される。pgvector 拡張入りイメージを採用し、意味検索の埋め込みベクトルを格納する。

項目設定
イメージpgvector/pgvector:pg16
コンテナ名agent-postgres
認証 / DBUSER=agent / PASSWORD=agent / DB=agent
ポート5432:5432
永続ボリュームpgdata:/var/lib/postgresql/data
ヘルスチェックpg_isready -U agent -d agent(interval 5s / timeout 5s / retries 5)
接続文字列postgresql://agent:agent@localhost:5432/agent

テーブルや拡張の作成はマイグレーションファイルではなく store.setup() によってアプリ起動時に自動で行われる(pgvector 拡張の有効化とテーブルの IF NOT EXISTS 作成)。起動 docker compose up -d で db を立ち上げ、アプリの lifespan で setup_store() が初期化する。backend/tests/test_memory.py も冒頭コメントで「docker compose up -d で db サービスが起動していること」を前提とし、実 Ollama 埋め込み + 実 PostgreSQL/pgvector で put/search のラウンドトリップ(test_put_and_search_roundtrip)を検証している。
