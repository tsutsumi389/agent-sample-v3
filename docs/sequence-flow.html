処理フロー / シーケンス

処理フロー

1リクエストがブラウザから FastAPI を経てオーケストレータ(親グラフ)に入り、記憶想起 → 計画 → 並列実行 → 評価 → (再計画ループ) → 最終回答生成 → 記憶化、と流れる過程を、orchestrator.py の制御フローと各段で発火する SSE イベントに沿って追跡する。

エンドツーエンド シーケンス

1リクエストの全体像を縦方向に示す。親グラフは recall_memory → planner → executor×N(並列) → evaluator → finalize をエッジで結び、評価結果に応じて planner へ戻る再計画ループを持つ。各ノードは writer() でカスタムイベントを発行し、service.py がそれを SSE として中継する。

補足 service.py は stream_mode=["custom","updates"] かつ subgraphs=True で astream を消費する。サブエージェント内部の writer() イベントも親ストリームへ浮上するため、planner/executor/evaluator の thinking や token がそのままフロントへ届く。

処理ステップ(時系列)

orchestrator.py のノード遷移と service.py の中継処理を時系列に列挙する。

  1. フロントの send() が POST /chat/stream を発行し、fetchEventSource で SSE 接続を開く(openWhenHidden:true)。reducer は start で状態を初期化しユーザーメッセージを追加する。
  2. run_agent_stream() が即座に status(node=orchestrator, state=running, "エージェント起動")を送出し、orchestrator.astream(inputs, stream_mode=["custom","updates"], subgraphs=True) を開始する。inputs は user_request と user_id。
  3. recall_memory: recall_memories(user_id, user_request) で長期記憶を想起。1件以上あれば memory イベントを発行し、retrieved_memories と iteration=0 を親 State へ書き戻す。
  4. planner_node: 要求・記憶・前回フィードバックのみを planner_agent.ainvoke に渡す。サブグラフ内で LLM をストリームし thinking を流し、plan イベント(tasks 配列)を発行。戻り値の plan を親 State へ書き戻す(JSON 解析に失敗した場合は要求そのものを1タスクにフォールバック)。
  5. assign_executors(条件エッジ): plan の各タスクを Send("executor", {instruction, task_id}) に変換してリスト返却し、N 個の executor を並列ファンアウトする。直前に status(node=orchestrator, "N件のタスクを並列実行へ振り分け")を送出。
  6. executor_node ×N(並列): 各 executor は割り当てられた単一 instruction / task_id のみで実行。task_update(running)→ 本文 token 逐次 → task_update(done)を発行し、task_results を operator.add リデューサ経由で1要素ずつ返す。
  7. evaluator_node(fan-in バリア): 全 executor 完了後に1回だけ起動。要求と task_results を evaluator_agent に渡し、evaluation(passed/feedback)イベントを発行。親 State に evaluation を書き戻し、iteration を +1 する。
  8. route_after_eval(条件エッジ): passed が真、または iteration >= max_iterations なら finalize へ。そうでなければ replan として planner へ戻る。
  9. finalize: status(finalize running)を送出し、全タスク結果を結合したプロンプトで統合 LLM をストリーム実行(emit_tokens=True で token を逐次)。final(answer)イベントを発行後、remember_async(user_id, user_request, answer) で背景記憶化(ホットパスをブロックしない)。final_answer を返す。
  10. 終端: astream 完了後、service.py が final "[DONE]" センチネルを送出。フロントは msg.data === "[DONE]" を検知して finish をディスパッチし、ストリーミング中の answer を確定メッセージへコミットして接続を abort する。

並列ファンアウトの詳細フロー

並列実行は LangGraph の Send と operator.add リデューサで実現している。planner の出力 plan の各タスクが個別の executor 起動へ展開され、全完了後に task_results がマージされてから evaluator へ進む。

   [planner_node]
        |  plan = [t1, t2, t3, ...]
        v
   add_conditional_edges("planner", assign_executors, ["executor"])
        |
        |  assign_executors(): plan の各 task を
        |  Send("executor", {instruction, task_id}) に変換しリスト返却
        v
   +---------------+   +---------------+   +---------------+
   | executor (t1) |   | executor (t2) |   | executor (t3) |   ← 並列起動
   |  instruction  |   |  instruction  |   |  instruction  |
   |  task_id 専任 |   |  task_id 専任 |   |  task_id 専任 |
   | return        |   | return        |   | return        |
   | task_results: |   | task_results: |   | task_results: |
   |   [{t1,...}]  |   |   [{t2,...}]  |   |   [{t3,...}]  |
   +-------+-------+   +-------+-------+   +-------+-------+
           \                  |                  /
            \                 |                 /
             v                v                v
        fan-in: ParentState.task_results は operator.add リデューサ
                各 executor の 1 要素リストを連結して安全にマージ
                       |
                       v   add_edge("executor","evaluator") = fan-in バリア
                  [evaluator_node]   ← 全 executor 完了後に 1 回だけ実行

Send によるファンアウト

assign_executors は通常のルータ関数ではなく Send オブジェクトのリストを返す。各 Send は executor ノードを独立に起動し、ペイロード {instruction, task_id} をその実行専用 State として注入する。executor は instruction 1件のみを知り、他タスクや会話履歴を持たない(コンテキスト分離)。

operator.add による fan-in

各 executor は task_results を1要素リストで返す。ParentState.task_results に operator.add リデューサが付与されているため、並列戻り値が衝突せずリスト連結でマージされる。add_edge("executor","evaluator") が fan-in バリアとなり、全 executor 完了後に evaluator が1回だけ起動する。

評価 → 再計画ループの分岐

評価後の分岐は route_after_eval(state) が担う。条件は evaluation.passed と累積 iteration(evaluator 通過ごとに +1)で決まる。max_iterations は settings.max_iterations。

                       [evaluator_node]
                  evaluation = {passed, feedback}
                  iteration += 1
                            |
                            v
                   route_after_eval(state)
                            |
        +-------------------+--------------------+
        |                                        |
  passed == True              passed == False かつ
   (合格)                       iteration < max_iterations
        |                       (不合格 / 上限未到達)
        |                                        |
        |          iteration >= max_iterations   |
        |           (上限到達 / 打ち切り)         |
        |                |                       |
        v                v                       v
   "finalize"       "finalize"               "replan"
        |                |                       |
        +-------+--------+                       v
                v                          [planner_node]
           [finalize]                feedback を反映して再計画
        最終回答を生成・記憶化         → 再び executor×N → evaluator
分岐条件遷移先
合格evaluation.passed が真finalize
上限到達iteration >= settings.max_iterations(passed が偽でも打ち切り)finalize
不合格上記いずれにも該当しない(passed が偽 かつ 上限未到達)replan → planner
再計画時の伝搬 planner_node は再入時に state["evaluation"]["feedback"] を取り出して planner_agent へ渡す。planner サブグラフは feedback がある場合「前回の計画は不十分でした。次の指摘を反映して再計画してください」とプロンプトへ追記し、計画を作り直す。
