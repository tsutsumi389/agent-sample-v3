API / SSE 仕様

FastAPI が公開する HTTP エンドポイントと、POST /chat/stream が返す Server-Sent Events（SSE）のイベント契約を定義する。バックエンドの main.py / service.py が発行するイベント名・JSON 構造と、フロントエンド（useAgentStream.ts / types.ts）が受信する型を突き合わせて記載する。

1. エンドポイント一覧

FastAPI アプリ（title="LangGraph Orchestrator Agent"）が公開するエンドポイントは 2 つ。中心は SSE ストリームを返す POST /chat/stream である。

メソッドパス概要レスポンス
GET /health ヘルスチェック。設定中のチャットモデル / 埋め込みモデル名を返す。 application/json
POST /chat/stream ユーザのメッセージをエージェントに渡し、実行過程と最終回答を SSE でストリーム配信する。 text/event-stream（EventSourceResponse）

GET /health のレスポンス例

{
  "status": "ok",
  "chat_model": "<settings.chat_model>",
  "embed_model": "<settings.embed_model>"
}

2. リクエストボディ（POST /chat/stream）

リクエストボディは main.py の ChatRequest（Pydantic BaseModel）で定義される。Content-Type: application/json で送信する。

フィールド型必須デフォルト説明
messagestr必須—ユーザの入力メッセージ。エージェントの user_request として渡される。
user_idstr任意"default-user"長期記憶の名前空間に使うユーザ識別子。
thread_idstr | None任意null会話スレッド ID。未指定（null）の場合はサーバ側で uuid.uuid4() を生成して使用する。
補足 フロントエンドの useAgentStream.ts は { message, user_id } のみを送信しており、thread_id は送らないため毎リクエストでサーバが新規スレッド ID を採番する。

State への写像

受け取ったリクエストは run_agent_stream(message, user_id, thread_id) に渡され、グラフ入力と config に展開される。

POST /chat/stream
  └─ ChatRequest { message, user_id, thread_id? }
        │
        ├─ inputs = { "user_request": message, "user_id": user_id }
        └─ config = { "configurable": { "user_id", "thread_id" } }
              │
              ▼
        orchestrator.astream(inputs, config,
            stream_mode=["custom","updates"], subgraphs=True)

3. SSE イベント仕様（章の中心）

SSE フレームは service.py の _sse(event, data) が生成する。event が SSE のイベント名、data は json.dumps(..., ensure_ascii=False) でシリアライズされた JSON 文字列になる。

def _sse(event, data):
    return {"event": event, "data": json.dumps(data, ensure_ascii=False)}

3.1 イベントの発生源

service.py は orchestrator.astream(stream_mode=["custom","updates"], subgraphs=True) を消費し、3 系統のフレームを生成する。

custom ノードの writer 発行

各ノードが writer() で発行した dict（type キーを持つ）をそのまま中継する。イベント名は chunk.get("type", "status")。subgraphs=True によりサブエージェント内部のイベントも親ストリームに浮上する。

updates ノード完了

ノード完了の {node: delta} を status イベント（state="node_complete"）へ変換する。

sentinel 終端

ストリーム末尾で必ず final イベントとして "[DONE]" を 1 回送る。SSE には終了通知が無いため、クライアントはこれで停止する。

注意 final イベントには 2 つの異なるペイロードが存在する。ノードが発行する { "type": "final", "answer": ... } と、ストリーム終端センチネルの文字列 "[DONE]" である。クライアントは msg.data === "[DONE]" を最優先で判定し、それ以外を JSON としてパースする。

3.2 イベント種別一覧

クライアントが識別する type は types.ts の AgentEvent 判別共用体で規定される。各イベントのペイロード構造を以下に示す。

SSE event 名type主なフィールド用途
statusstatusnode, state, messageノードの稼働状況。state は running / node_complete。
thinkingthinkingagent, task_id?, contentサブエージェントの思考の逐次出力（追記で連結）。
tokentokenagent, task_id?, content最終回答（finalize）本文トークンの逐次出力。
planplantasks[]プランナが生成したタスク一覧。
task_updatetask_updatetask_id, status, label個別タスクの状態遷移。
evaluationevaluationpassed, feedback評価ノードの合否とフィードバック。
memorymemoryitems[]検索された長期記憶の一覧。
finalfinalanswer または文字列 "[DONE]"最終回答（ノード発行）／ストリーム終端センチネル（終端）。
errorerrormessage実行中の例外。service.py の except で送出。

3.3 ペイロード詳細と JSON 例

status

ストリーム開始時に service.py がまず node="orchestrator" / state="running" / message="エージェント起動" を送る。また updates モードのノード完了を state="node_complete"（message は空文字）に変換して送る。クライアントは state==="running" のときだけ currentNode を更新する。

event: status
data: {"type":"status","node":"orchestrator","state":"running","message":"エージェント起動"}

event: status
data: {"type":"status","node":"planner","state":"node_complete","message":""}

plan

tasks は各要素が { id, instruction, parallel_group }。parallel_group は並列実行グループ番号。クライアントは instruction を label、parallel_group を parallelGroup にマッピングし、初期 status を pending とする。

event: plan
data: {"type":"plan","tasks":[
  {"id":"t1","instruction":"資料を収集する","parallel_group":0},
  {"id":"t2","instruction":"要点を整理する","parallel_group":1}
]}

task_update

個別タスクの状態遷移。status は types.ts の TaskStatus（pending / running / done / error）。クライアントは task_id で対象タスクを特定し、label が空でなければ更新する。

event: task_update
data: {"type":"task_update","task_id":"t1","status":"running","label":"資料を収集する"}

thinking

サブエージェントの思考ストリーム。クライアントは直前のエントリと agent および task_id（未指定時は null）が一致する場合、content を末尾に追記してまとめる。

event: thinking
data: {"type":"thinking","agent":"planner","task_id":null,"content":"まず必要な"}

event: thinking
data: {"type":"thinking","agent":"planner","task_id":null,"content":"情報を洗い出す"}

token

finalize 段階の最終回答本文を 1 トークンずつ送る。クライアントは answer 文字列に content を逐次追記する。

event: token
data: {"type":"token","agent":"finalize","task_id":null,"content":"こんに"}

event: token
data: {"type":"token","agent":"finalize","task_id":null,"content":"ちは"}

evaluation

評価ノードの結果。passed（合否）と feedback（再計画用フィードバック）を持つ。

event: evaluation
data: {"type":"evaluation","passed":false,"feedback":"根拠の引用が不足している"}

memory

検索された長期記憶の一覧を文字列配列 items で送る。クライアントは memories を丸ごと置き換える。

event: memory
data: {"type":"memory","items":["前回はTypeScriptを利用","東京在住"]}

final / 終端センチネル

ノードが発行する final は answer を持ち、クライアントは answer が空でなければ最終回答に反映する。ストリーム末尾の終端センチネルは JSON ではなく文字列 "[DONE]" であり、クライアントは msg.data === "[DONE]" を検出した時点でストリームを停止・確定する。

event: final
data: {"type":"final","answer":"最終的な回答テキスト"}

event: final
data: "[DONE]"

error

実行中に例外が発生した場合、service.py が error イベントを送ってからループを終える（その後も終端センチネルは送られる）。

event: error
data: {"type":"error","message":"<例外メッセージ>"}

4. 利用例

4.1 curl（SSE をそのまま観察）

POST /chat/stream に JSON ボディを送り、生の SSE フレームを標準出力で確認する例。

curl -N -X POST http://localhost:8000/chat/stream \
  -H "Content-Type: application/json" \
  -d '{"message":"自己紹介して","user_id":"default-user"}'
tip -N（--no-buffer）を付けないと SSE がバッファリングされ、逐次受信が観察できない。

4.2 fetch + EventSource 的な受信（フロント実装に即して）

実装では標準 EventSource ではなく、POST ボディを送れる @microsoft/fetch-event-source の fetchEventSource を使う。useAgentStream.ts の send() に準ずる最小例を示す。

import { fetchEventSource } from "@microsoft/fetch-event-source";

const ctrl = new AbortController();
await fetchEventSource("/chat/stream", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ message, user_id: "default-user" }),
  signal: ctrl.signal,
  openWhenHidden: true, // バックグラウンドタブでも継続
  onmessage(msg) {
    if (msg.data === "[DONE]") { ctrl.abort(); return; } // 終端
    if (!msg.data) return;
    const ev = JSON.parse(msg.data); // AgentEvent
    // ev.type で分岐（status / plan / thinking / token ...）
  },
  onerror(err) { throw err; }, // 自動リトライを止める
});
受信契約 onmessage は "[DONE]" を最優先で判定し、空 data をスキップ、それ以外を JSON.parse して AgentEvent として処理する。パースできないフレームは無視される。onerror で例外を再 throw することでライブラリの自動リトライを停止している。

4.3 クライアント側の中断とサーバ側の打ち切り

クライアント                          サーバ (main.py)
   │  AbortController.abort()             │
   │ ───────────────────────────────────▶│ request.is_disconnected() == True
   │                                      │   → event_source ループを break
   │  ("[DONE]" 受信でも abort)            │   (無駄な計算を停止)

main.py の event_source() は各イベント送出前に request.is_disconnected() を確認し、クライアント切断時はループを break して以降の計算を打ち切る。

5. CORS / ミドルウェア設定

main.py は CORSMiddleware を 1 つだけ登録している。許可オリジンは設定値 settings.frontend_origin の単一オリジン。

項目設定値
allow_origins[settings.frontend_origin]（単一オリジン）
allow_credentialsTrue
allow_methods["*"]
allow_headers["*"]
 
app.add_middleware(
    CORSMiddleware,
    allow_origins=[settings.frontend_origin],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

ライフサイクル（lifespan）

FastAPI の lifespan コンテキストで、起動時に埋め込み次元チェック（assert_embed_dims）と PostgreSQL 記憶ストア初期化（setup_store）を行い、終了時に teardown_store を呼ぶ。いずれの初期化も失敗時は warning ログを出して継続する（記憶検索／永続化が無効になりうる）。

startup
  ├─ assert_embed_dims()   失敗 → warning（記憶検索が動かない可能性）
  └─ setup_store()         失敗 → warning（記憶が永続化されません）
        │  yield（リクエスト処理）
        ▼
shutdown
  └─ teardown_store()
