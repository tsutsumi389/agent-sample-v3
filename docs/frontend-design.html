フロントエンド設計

React + TypeScript + Vite で構築したシングルページ UI。SSE（Server-Sent Events）でバックエンドの実行状況をリアルタイムに受信し、チャット欄とエージェント稼働パネルを同時に更新する。本章ではコンポーネント構成・useAgentStream フックによるイベント駆動の状態管理・型定義・可視化 UX を、ソースコードに基づき詳説する。

1. 技術スタックとエントリポイント

ビルド / 配信

Vite（@vitejs/plugin-react）。開発サーバはポート 5173。/chat と /health はバックエンド（http://localhost:8000）へ changeOrigin: true でプロキシする（vite.config.ts）。

SSE クライアント

@microsoft/fetch-event-source を採用。標準 EventSource と異なり POST + JSON ボディでの SSE 受信が可能。

マウント

main.tsx が #root に <React.StrictMode> でラップした <App /> をレンダリング。スタイルは styles.css を読み込み。

2. コンポーネント構成

App は単一フックの状態を 2 つの主要パネルへ配り、画面を左右 2 カラム（.layout）に分けて構成する。

3. useAgentStream フック（中核の状態管理）

useReducer でアプリ全状態（ChatState）を一元管理し、SSE 受信を useRef 保持の AbortController で制御する。send / stop は useCallback で安定化している。

3.1 アクションとライフサイクル

reducer は 4 種の Action を処理する。start で状態をリセットしユーザメッセージを追加、event で SSE イベントを applyEvent に委譲、finish でストリーム中の answer を確定メッセージへコミット、fail でエラー文言を設定する。

type Action =
  | { kind: "start"; message: string }
  | { kind: "event"; event: AgentEvent }
  | { kind: "finish" }
  | { kind: "fail"; message: string };

case "start":
  return {
    ...initialState,
    messages: [...state.messages, { role: "user", content: action.message }],
    running: true,
    statusMessage: "起動中…",
  };
case "finish": {
  // ストリーミングした answer を確定メッセージにコミット
  const msgs = state.answer
    ? [...state.messages, { role: "assistant" as const, content: state.answer }]
    : state.messages;
  return { ...state, running: false, currentNode: null, messages: msgs, answer: "" };
}
設計ポイント start は ...initialState を展開してタスク・思考・評価などを毎ターン破棄する。前ターンの messages のみ引き継ぎ、新規実行ごとに稼働パネルをクリーンな状態から再構築する。

3.2 SSE 受信ループ

fetchEventSource("/chat/stream", ...) を POST で開く。openWhenHidden: true によりバックグラウンドタブでも実行を継続する。各フレームは onmessage で処理し、[DONE] を受けたら finish を発行して abort() する。

onmessage(msg) {
  if (msg.data === "[DONE]") {
    dispatch({ kind: "finish" });
    ctrl.abort();
    return;
  }
  if (!msg.data) return;
  try {
    const parsed = JSON.parse(msg.data) as AgentEvent;
    dispatch({ kind: "event", event: parsed });
  } catch {
    // パースできないフレームは無視
  }
},
onerror(err) {
  dispatch({ kind: "fail", message: String(err) });
  throw err; // 自動リトライを止める
},
エラー設計 onerror 内で throw err することで fetch-event-source の自動リトライを無効化し、無限再接続を防いでいる。stop() は ctrlRef.current?.abort() 後に finish を発行してユーザ起点の停止を実装する。

3.3 イベント種別ごとの状態更新（applyEvent）

applyEvent は AgentEvent.type で分岐し、各イベントを ChatState の対応フィールドへ反映する。すべて新オブジェクトを返すイミュータブル更新。

typeペイロード主要フィールド状態更新
statusnode, state("running"/"node_complete"), message?state === "running" のとき currentNode を更新。message があれば statusMessage を更新。
thinkingagent, task_id?, contentthoughts へ追加。直前エントリと agent / taskId が一致する場合は content を末尾連結してまとめる。
tokenagent, task_id?, contentanswer に content を逐次追記（finalize 本文のストリーミング）。
plantasks[]（id, instruction, parallel_group）tasks を再構築。instruction→label、parallel_group→parallelGroup に写像し、全件 status: "pending"。
task_updatetask_id, status, label該当 id のタスクの status を更新。label があれば差し替え。
evaluationpassed, feedbackevaluation に { passed, feedback } を設定。
memoryitems[]memories を items で置換。
finalansweranswer を確定値で上書き（空なら既存値を維持）。
errormessagestatusMessage を エラー: {message} に設定。

連続する思考のマージは、トークン単位で届く思考断片を 1 エントリに集約するための処理である。

case "thinking": {
  // 同一 agent/task の連続する思考は末尾に追記してまとめる
  const last = state.thoughts[state.thoughts.length - 1];
  if (last && last.agent === ev.agent && last.taskId === (ev.task_id ?? null)) {
    const merged = [...state.thoughts];
    merged[merged.length - 1] = { ...last, content: last.content + ev.content };
    return { ...state, thoughts: merged };
  }
  return {
    ...state,
    thoughts: [...state.thoughts, { agent: ev.agent, taskId: ev.task_id ?? null, content: ev.content }],
  };
}

4. 型定義（types.ts）

バックエンドの SSE イベントとフロントの状態モデルを TypeScript の型で表現する。

型定義 / フィールド用途
TaskStatus"pending" | "running" | "done" | "error"タスクの進行状態。
AgentTaskid, label, status: TaskStatus, parallelGroup: numberチェックリスト 1 行分。
ThoughtEntryagent, taskId?: string | null, content思考ログ 1 エントリ。
ChatMessagerole: "user" | "assistant", content確定チャット履歴。
EvalInfopassed: boolean, feedback: string評価結果。
ChatStaterunning, currentNode, statusMessage, tasks, thoughts, memories, evaluation, answer, messagesUI 全体の単一ステート。
AgentEvent判別共用体（type 判別）: status / thinking / token / plan / task_update / evaluation / memory / final / errorSSE data ペイロードの型。
命名の橋渡し SSE 側は task_id / parallel_group（スネークケース）だが、フロント状態は taskId / parallelGroup（キャメルケース）に変換される。この写像は applyEvent が担う。

5. リアルタイム可視化 UX

同一の ChatState を左右のパネルが購読することで、チャットと稼働パネルが同時に更新される。SSE イベントが届くたびに reducer が新状態を返し、React の再レンダリングで両パネルが反映される。

チャット側（ChatWindow）

running && answer のときは token で追記される回答にカーソル ▍ を添えてストリーミング表示。running だが answer 未着のときはスピナー＋statusMessage。messages / answer の変化で末尾へ自動スクロール。

現在ノード表示

status イベントの node を NODE_LABELS で日本語化しバッジ表示。実行中は live、停止後は「待機中」(idle)。statusMessage を補足表示。

タスクチェックリスト

plan でタスク一覧が生成され、task_update で個別状態が遷移。parallelGroup が同じタスクは「グループN（同時実行）」としてまとめ、並列実行を視覚化する。

思考フィード

thinking を時系列に表示。同一 agent/task の連続断片はマージされ、長文思考が 1 ブロックとして滑らかに伸びる。新規追加ごとに自動スクロール。

1 ターンのイベント到着順と UI 反映を時系列で示す。

   │   ┌──────────────┐         ──────────              ──────────────────
   │   │ user msg 追加 │         user 吹き出し            「起動中…」
   ▼   └──────────────┘
 SSE  status(memory)            spinner + statusMessage  バッジ: 記憶想起
 SSE  memory(items)             ─                        想起した記憶リスト

状態の単一源泉 UI 状態は useAgentStream の ChatState ただ 1 つ。コンポーネントはローカル状態を最小限（Composer の入力テキスト、自動スクロール用 ref）に留め、表示は全て props 経由の純粋関数的レンダリングで統一されている。
