詳細設計 / Backend

バックエンド設計

FastAPI を入口に、LangGraph のオーケストレータ親グラフと 3 つのサブエージェント（Planner / Executor / Evaluator）を独立コンパイル済みグラフとして構成する。graph.astream の出力を SSE へ変換し、各エージェントが writer() で発行するイベントをリアルタイムにクライアントへ中継する。本章は backend/app 配下の実装に即した詳細設計をまとめる。

1. モジュール構成

backend/app 配下の各モジュールの責務は次のとおり。

モジュール主な定義責務
main.py app, lifespan, ChatRequest, /health, /chat/stream FastAPI エントリポイント。起動時に埋め込み次元チェックと PostgreSQL 記憶ストア初期化を行い、POST /chat/stream で SSE を返す。CORS は settings.frontend_origin のみ許可。
service.py run_agent_stream, _sse, _unpack LangGraph の astream 出力（custom / updates）を SSE イベントへ変換するアダプタ層。
orchestrator.py build_orchestrator, orchestrator, 各ノード関数 親グラフ。記憶想起 → 計画 → 並列実行 → 評価 → 最終化のフローと、評価後の再計画ルーティングを定義。
llm.py get_chat_llm, get_embeddings ChatOllama / OllamaEmbeddings のファクトリ。reasoning 値のバージョン差異吸収と num_ctx 明示を担う。
config.py Settings, settings 環境変数 / .env からの設定読み込み（pydantic-settings）。
agent_utils.py safe_writer, stream_agent, parse_json, _as_text エージェント共通ユーティリティ。LLM ストリーミング送出と JSON 抽出を提供。
agents/planner.py planner_agent, _plan, _normalize ユーザー要求を並列実行可能な独立タスクへ分解するサブエージェント。
agents/executor.py executor_agent, _execute 単一 instruction を実行するサブエージェント。Send により並列起動される。
agents/evaluator.py evaluator_agent, _evaluate, _normalize 要求と結果から合否を判定し、不合格なら再計画フィードバックを返すサブエージェント。
設計方針 各サブエージェントは独立コンパイル済みの StateGraph であり、親グラフと State キーを共有しない。ブリッジノードが「明示的な指示だけ」を渡し、戻り値の必要なキーだけを親 State に書き戻すことで、エージェント間でコンテキスト（会話履歴）を一切引き継がない。

2. サブエージェント詳細

3 つのサブエージェントはいずれも単一ノードの最小グラフ（START → 処理ノード → END）として _build() でコンパイルされ、モジュールロード時にインスタンス化される。

2.1 Planner エージェント

入力 PlannerState: request（ユーザー要求）, memories（関連記憶の配列）, feedback（再計画時の指摘）

出力 {"plan": list[Task]} — Task は id / instruction / parallel_group を持つ

役割 要求を互いに独立して並列実行できる具体的なサブタスクに分解する。

プロンプト方針: system プロンプトで「各タスクは 1 つの明確な指示文」「並列実行可能なタスクには同じ parallel_group 番号（0始まり）」「タスクは多くても 5 個、単純なら 1 個」を指示。出力は {"tasks":[{"id":"t1","instruction":"...","parallel_group":0}]} の JSON のみを要求する。human 側には要求本文に加え、memories があれば「関連する過去の記憶」、feedback があれば「前回の計画は不十分でした。次の指摘を反映して再計画してください」を追記する。

JSON 抽出 / フォールバック: stream_agent で得た本文を parse_json → _normalize で list[Task] 化する。_normalize は tasks が非リスト/空、要素が dict でない、instruction が空のものを除外し、id 欠落時は f"t{i+1}" を補完する。正規化結果が空なら、要求そのものを単一タスク（id="t1", parallel_group=0）にするフォールバックが働く。

2.2 Executor エージェント

入力 ExecutorState: instruction, task_id

出力 {"result": content}（実行成果物の本文全体）

役割 与えられた 1 つの instruction だけを忠実に実行する。Send により複数の Executor が並列起動される。

プロンプト方針: system プロンプトで「与えられた 1 つのタスクだけを忠実に実行」「余計な前置きをせず簡潔かつ具体的に」「1 タスク専任であり他のタスクや会話の文脈は知らない」を明示。human には instruction をそのまま渡す。

ストリーミング: 実行前に task_update（status="running", label=instruction[:60]）を発行し、stream_agent を emit_tokens=True で呼んで本文トークンを逐次送出、完了後に task_update（status="done"）を発行する。JSON 抽出は行わず本文をそのまま結果とする。

2.3 Evaluator エージェント

入力 EvaluatorState: request, results（タスク結果の配列）

出力 {"evaluation": Evaluation} — Evaluation は passed（bool）/ feedback（str）

役割 元の要求と各タスク結果を読み、要求が十分に満たされているか判定。不十分なら再計画用のフィードバックを返す。

プロンプト方針: system プロンプトで品質評価者として判定を求め、出力は {"passed": true/false, "feedback": "..."} の JSON のみを要求する（passed が true なら feedback は空文字でよい）。human には要求本文と、各結果を [タスク {task_id}] 指示: ... / 結果: ... 形式で連結したテキストを渡す。

JSON 抽出 / フォールバック: parse_json の結果が空でなければ _normalize（passed を bool 化、欠落時は True / feedback を str 化）し、空なら empty_evaluation() を使う。判定後に evaluation イベントと、合否に応じた node_complete ステータス（「合格」/「再計画が必要」）を発行する。

3. LLM ファクトリ（llm.py）

get_chat_llm は ChatOllama を生成するファクトリで、@lru_cache(maxsize=None) により引数（streaming, temperature）ごとにキャッシュされる。ChatOllama はステートレス（astream は渡したメッセージのみで推論）なため、全エージェントで同一インスタンスを共有しても会話履歴は混ざらない。

@lru_cache(maxsize=None)
def get_chat_llm(*, streaming: bool = True, temperature: float = 0.0) -> ChatOllama:
    base = dict(
        model=settings.chat_model,
        base_url=settings.ollama_base_url,
        temperature=temperature,
        num_ctx=settings.num_ctx,
        streaming=streaming,
    )
    # reasoning パラメータをバージョン差異に強い形で適用する。
    for reasoning in (settings.reasoning_effort, True, None):
        try:
            kwargs = dict(base)
            if reasoning is not None:
                kwargs["reasoning"] = reasoning
            return ChatOllama(**kwargs)
        except Exception:  # noqa: BLE001 - バージョン非対応値を順に試す
            continue
    return ChatOllama(**base)

reasoning_effort

settings.reasoning_effort（既定 "medium"、low / medium / high）→ True → None の順にフォールバックする。gpt-oss では reasoning="medium" を渡すと思考（CoT）が content ではなく additional_kwargs['reasoning_content'] に分離され、UI の「思考」表示に使える。古い langchain-ollama は文字列値を受け付けず bool_parsing エラーになるため、段階的に試行する。

num_ctx

settings.num_ctx（既定 8192）を明示的に渡す。明示しないと Ollama がコンテキストを約 4096 トークンに切り詰めてしまうため、必ず指定する。

get_embeddings

OllamaEmbeddings を @lru_cache(maxsize=1) で単一生成。model=settings.embed_model（既定 nomic-embed-text）、base_url=settings.ollama_base_url。長期記憶のベクトル化に用いる。

4. 共通ユーティリティ（agent_utils.py）

4.1 safe_writer

グラフ実行コンテキスト内なら LangGraph の get_stream_writer() を返し、コンテキスト外（ルータ関数やユニットテスト）では no-op（lambda *_args, **_kwargs: None）を返すラッパー。例外を握りつぶすことで、どこから呼んでも安全に writer(...) を呼べる。

4.2 stream_agent

LLM を astream し、本文と思考を逐次イベント送出しながら (本文全体, 思考全体) のタプルを返す中核関数。

async def stream_agent(
    llm: BaseChatModel,
    messages: list[BaseMessage] | list[tuple[str, str]],
    *,
    agent: str,
    writer,
    task_id: str | None = None,
    emit_tokens: bool = False,
) -> tuple[str, str]:
    ...
    async for chunk in llm.astream(messages):
        text = _as_text(getattr(chunk, "content", ""))
        if text:
            content_parts.append(text)
            if emit_tokens:
                writer({"type": "token", "agent": agent, "task_id": task_id, "content": text})

        extra = getattr(chunk, "additional_kwargs", {}) or {}
        reasoning = extra.get("reasoning_content") or extra.get("reasoning")
        if reasoning:
            reasoning_parts.append(reasoning)
            writer({"type": "thinking", "agent": agent, "task_id": task_id, "content": reasoning})

    return "".join(content_parts), "".join(reasoning_parts)
  • token イベント: 本文差分（emit_tokens=True のときのみ送出）。Executor と finalize で有効。Planner / Evaluator は本文を送らず JSON だけ取り出す。
  • thinking イベント: reasoning_content または reasoning の差分。バージョン差異に両対応。
  • _as_text が content を str / list（テキストブロック）いずれでも文字列へ正規化する。

4.3 parse_json

不安定な LLM の構造化出力から JSON オブジェクトを頑健に取り出す。コードフェンス除去 → 直接 json.loads → 最初の {...} ブロック抽出、の順に試し、いずれも失敗すれば {} を返す。フェンスは正規表現 ```(?:json)?\s*(.*?)\s*```（DOTALL）で検出する。

5. ストリーム変換ロジック（service.py）

run_agent_stream が orchestrator.astream を消費して SSE イベントの非同期ジェネレータを返す。stream_mode=["custom", "updates"] と subgraphs=True を指定することで、サブエージェント内部の writer イベントも親ストリームへ浮上させている。

async for item in orchestrator.astream(
        inputs,
        config=config,
        stream_mode=["custom", "updates"],
        subgraphs=True,
):
    mode, chunk = _unpack(item)

    if mode == "custom" and isinstance(chunk, dict):
        event = chunk.get("type", "status")
        yield _sse(event, chunk)

    elif mode == "updates" and isinstance(chunk, dict):
        for node in chunk:
            yield _sse("status",
                {"type": "status", "node": node, "state": "node_complete", "message": ""})

_unpack による形の正規化

subgraphs=True + 複数 stream_mode では出力が (namespace, mode, chunk) の 3 要素、単一モードでは (namespace, chunk) の 2 要素になりうる。_unpack は要素数で分岐し、3 要素なら (mode, chunk)、2 要素ならそのまま、それ以外は ("custom", item) として正規化する。

イベント変換

custom: 各ノードが writer() で発行した dict をそのまま中継し、type を SSE の event 名にする（status / token / thinking / plan / task_update / evaluation / memory / final など）。updates: ノード完了の {node: delta} を node_complete ステータスへ変換する。

処理の前後で制御イベントを補う。冒頭で orchestrator / running の「エージェント起動」ステータスを送り、ループを try/except で囲んで例外時は error イベントを送出、最後に SSE には終了通知が無いため終端センチネル _sse("final", "[DONE]") を必ず送る。_sse はデータを json.dumps(..., ensure_ascii=False) で文字列化し {"event":..., "data":...} 形にする。

なお main.py の event_source はループ内で request.is_disconnected() を確認し、クライアント切断時に実行を打ち切って無駄な計算を止める。

6. 評価 → 再計画ループ

親グラフのフローは次のとおり。evaluator 後の条件エッジ route_after_eval が、合否と反復回数で finalize か replan（= planner へ戻る）かを決める。

  START
    │
    ▼
 recall_memory ── retrieved_memories, iteration=0
    │
    ▼
 planner ◀───────────────┐ (replan: feedback を反映して再計画)
    │ assign_executors    │
    │ (Send 並列ファンアウト) │
    ▼                     │
 executor × N ── task_results (operator.add でマージ)
    │ (fan-in バリア)      │
    ▼                     │
 evaluator ── evaluation, iteration += 1
    │                     │
    ▼ route_after_eval    │
  ┌─────────────┐         │
  │ passed?      │── No ──┘  (かつ iteration < MAX_ITERATIONS)
  │ or 上限到達?  │
  └─────┬───────┘
        │ Yes
        ▼
    finalize ── final_answer, remember_async
        │
        ▼
       END
def route_after_eval(state: ParentState) -> str:
    evaluation = state.get("evaluation") or {}
    if evaluation.get("passed", True) or state.get("iteration", 0) >= settings.max_iterations:
        return "finalize"
    return "replan"
MAX_ITERATIONS 上限 反復回数 iteration は recall_memory で 0 に初期化され、evaluator_node が完了するたびに +1 される。route_after_eval は passed が真（既定は True、評価が取れない場合は合格扱い）か、iteration が settings.max_iterations（既定 2）以上なら finalize へ。それ以外は replan として planner に戻り、evaluation.feedback を渡して再計画する。これにより評価ループが無限に回ることを防ぐ。

再計画時、planner_node は (state.get("evaluation") or {}).get("feedback", "") を取り出して Planner へ渡す。Planner はこの feedback を human プロンプトに反映し、指摘を踏まえた新しい計画を生成する。最終化（finalize）では各タスク結果を連結したテキストを FINALIZE_SYSTEM とともに stream_agent（emit_tokens=True）へ渡して最終回答を生成し、final イベント送出後に remember_async で会話を背景記憶化する（ホットパスをブロックしない）。
