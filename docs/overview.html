概観 / Overview

システム概要

ローカルの Ollama (gpt-oss) 上で動作する、オーケストレータ + プランナー / エグゼキューター(並列) / エヴァリュエーター 構成の AI エージェント。 各エージェントの稼働状況と思考内容を、フロントエンドのチャット画面へ Server-Sent Events (SSE) でリアルタイム表示する。

1. このシステムは何か

本システムは、ユーザーの要求を 1 つの大規模言語モデル呼び出しで処理するのではなく、 役割の異なる複数のエージェントを LangGraph のグラフ上で連携させて解く オーケストレータ型 (orchestrator-worker) の AI エージェントである。 言語モデルはクラウド API ではなく、ローカルで起動した Ollama の gpt-oss:20b を用いる。

処理は次の親グラフ(オーケストレータ)で流れる。

  ユーザー
    │  (React + SSE)
    ▼
  FastAPI  /chat/stream
    │
    ▼  LangGraph 親グラフ (オーケストレータ)
  START → recall_memory ─→ planner ─┬─[Send 並列]→ executor(t1)
                                    ├──────────────→ executor(t2)   ← 並列実行
                                    └──────────────→ executor(t3)
                                           │ (fan-in)
                                           ▼
                                       evaluator ──┬─ 合格/上限 → finalize → END
                                                   └─ 不合格      → planner(再計画)
         │
         ▼ 背景で記憶化 (LangMem)
    PostgreSQL + pgvector + nomic-embed-text(768次元)
  • recall_memory: 長期記憶 (LangMem) から関連する記憶を呼び戻す。
  • planner: 要求をタスクへ分解する。
  • executor: 分解されたタスクを Send API で並列ファンアウトして実行する。
  • evaluator: 結果を評価し、不合格なら planner へ再計画として戻す(MAX_ITERATIONS 上限あり)。
  • finalize: 最終回答を確定する。背景で会話を記憶化する。

2. 解決する課題と特徴

コンテキストの肥大化

各エージェント(Planner / Executor / Evaluator)は独立してコンパイルされた StateGraph で、 親グラフと State キーを共有しない。会話履歴を引き継がず、毎回まっさらなコンテキストで起動するため、 プロンプトの肥大化を抑える。

逐次処理による遅延

Planner が出力した複数タスクを、オーケストレータが Send API で並列ファンアウトする。 複数 Executor の結果は Annotated[list, operator.add] のリデューサで安全にマージする。

エージェントの不透明さ

各ノードが get_stream_writer() で status / thinking / plan / task_update / token / evaluation を発行し、 SSE 経由でフロントの「稼働パネル」とチャットを同時更新。今どのエージェントが何を考えているかを可視化する。

セッションを跨いだ記憶

会話を LangMem で記憶化し、PostgreSQL + pgvector に永続化する。 プロセス再起動後も記憶が残り、recall_memory で意味検索により呼び戻せる。

4. 主要な設計思想

コンテキスト分離 各エージェントは独立コンパイルされた StateGraph として動き、親グラフと State キーを一切共有しない。 オーケストレータはブリッジノードから明示的な指示文だけを渡し、戻り値の必要なキーだけを読み戻す。 会話履歴は引き継がれず、各エージェントは毎回まっさらなコンテキストで起動する。
並列ファンアウト Planner が出力したタスクを Send API で並列に Executor へ展開し、結果は Annotated[list, operator.add] のリデューサで安全にマージする。
リアルタイムストリーミング FastAPI が astream(stream_mode=["custom","updates"], subgraphs=True) を SSE に変換し、 React の useAgentStream フックがチャットと稼働パネルを同時更新する。 gpt-oss の思考 (reasoning) は reasoning_content に分離され、思考ログに流れる。
既知の制約 DB が未起動だと記憶検索 / 保存はスキップされる(チャット自体は動作)。 単一 Ollama インスタンスへの並列 Executor は OLLAMA_NUM_PARALLEL 次第でモデルサーバ側で直列化されうる。 gpt-oss の structured output は不安定なため、Planner / Evaluator は寛容な JSON 抽出 + フォールバックで対処する。
